# Layer Analysis: Security
> Generated by: WU-6 (Sequential Fallback) | Date: 2026-03-14

## Summary
- Server-side code uses the public Supabase anon key for server operations (risk of privilege escalation).
- Row-Level Security (RLS) is comprehensive, but SECURITY DEFINER RPCs lack explicit search_path hardening.
- Error responses leak internal error.message values in multiple API routes.
- Local dev environment mounts the Docker socket for LocalStack and exposes PostGIS on 0.0.0.0, increasing host attack surface.
- CSP includes `unsafe-eval` and `unsafe-inline` for scripts (Cesium requirement documented; weakens XSS posture).
- Custom impersonation token HMAC implementation exists (requires review and key management hygiene).

## Findings
- Server anon key used server-side: src/lib/supabase/server.ts — createServerClient uses NEXT_PUBLIC_SUPABASE_ANON_KEY (evidence in analysis files). This allows accidental misuse of the public key in privileged flows.
- SECURITY DEFINER RPCs do tenant checks but do not set search_path; recommendation in analysis_04_backend.md to add `SET search_path = public` (supabase/migrations/*sql).
- Error leakage pattern: many API routes use `catch (error) { return NextResponse.json({ error: error.message }, { status: 500 }) }` (see src/app/api/**/route.ts examples referenced in analysis_03_services_api.md).
- Docker socket mount and PostGIS port exposure: docker-compose.yml exposes PostGIS `"5432:5432"` and binds `/var/run/docker.sock:/var/run/docker.sock` for LocalStack (analysis_05_devops.md lines 36–48).
- CSP exceptions: next.config.ts includes `"script-src 'self' 'unsafe-eval' 'unsafe-inline' https://cesium.com"` (analysis_05_devops.md line 81).

## Recommendations (Prioritised)
P0 — Must fix before M17 ships
1. Stop using NEXT_PUBLIC_SUPABASE_ANON_KEY for server-side privileged operations. Use one of:
   - Migrate server code to use SUPABASE_SERVICE_ROLE_KEY for admin flows and protect it with a secrets manager; OR
   - Migrate to @supabase/ssr which provides proper server context handling (analysis_03_services_api.md Finding 3.2).
   - Files: src/lib/supabase/server.ts, src/middleware.ts
   - Effort: M | Risk: MED
   - Acceptance: No server code references NEXT_PUBLIC_SUPABASE_ANON_KEY for admin actions; verified by grep `NEXT_PUBLIC_SUPABASE_ANON_KEY`.

2. Harden SECURITY DEFINER functions by setting `SET search_path = public` at function start (supabase/migrations/* — see analyze_area and search_properties SQL files).
   - Effort: S | Risk: LOW

P1 — Should fix this sprint
3. Stop returning raw error.message to clients. Replace with generic messages and send detailed errors to server logs/monitoring (monitor.trackError()).
   - Files: src/app/api/**/route.ts (all mutation routes)
   - Effort: S | Risk: LOW

4. Remove Docker socket mount from LocalStack and restrict PostGIS bind address to localhost in docker-compose.yml (change `"5432:5432"` → `"127.0.0.1:5432:5432"`).
   - Effort: S | Risk: LOW

P2 — Nice to have
5. Document CSP `unsafe-eval` as an explicit, reviewed exception for Cesium and remove `unsafe-inline` by using nonces for scripts where possible.
6. Rotate and store impersonation token signing keys in a secrets manager. Consider replacing custom HMAC token implementation with a vetted JWT library and include key rotation rules.

## Rollback / Mitigation

- For CSP changes, if Cesium breaks, reintroduce `unsafe-eval` behind a feature flag and schedule a Cesium-worker isolation plan.
