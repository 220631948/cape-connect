# Layer Analysis: Services & API Layer
> Generated by: WU-3 (Coordinator Fallback) | Date: 2026-03-14

## Summary
- 22 API routes in the worktree (8 admin, 3 analysis, 2 flights, 3 invitations, 1 search, 1 suburbs, 1 valuation, 1 zoning) — only 4 at root
- Zero input validation — all routes use raw `await request.json()` without zod/AJV
- All server-side Supabase clients use `NEXT_PUBLIC_SUPABASE_ANON_KEY` (by design — RLS + SECURITY DEFINER RPCs handle escalation)
- Three-tier fallback (LIVE→CACHED→MOCK) properly implemented on data routes (flights, zoning, suburbs, valuation)
- Rate limiting in middleware (60 req/min per IP) covers all API routes

## Findings

### Finding 3.1: No Input Validation on Any API Route
- **Severity:** HIGH
- **Category:** Security
- **Files Affected:** All 22 `src/app/api/**/route.ts` files
- **Description:** Every API route that accepts POST/PUT body parses it with `await request.json()` without schema validation. This allows malformed data, type confusion, and potential injection attacks.
- **Evidence:** `src/app/api/admin/assign-role/route.ts`: `const { userId, role } = await request.json()` — no type checking or validation
- **Recommendation:** Install zod and add request schemas to every mutation route. Example: `const schema = z.object({ userId: z.string().uuid(), role: z.enum(['VIEWER','ANALYST',...]) }); const body = schema.parse(await request.json());`
- **Effort:** M | **Risk:** LOW
- **Acceptance Criteria:** Invalid payloads return 400 with descriptive errors; valid payloads work unchanged
- **Rollback:** Remove zod schemas and revert to raw parsing

### Finding 3.2: Deprecated `@supabase/auth-helpers-nextjs`
- **Severity:** HIGH
- **Category:** Maintainability
- **Files Affected:** `src/lib/supabase/server.ts`, `src/lib/supabase/client.ts`, `src/middleware.ts`
- **Description:** `@supabase/auth-helpers-nextjs@^0.15.0` is deprecated. Supabase recommends `@supabase/ssr` which provides better Next.js 15 App Router support with proper cookie handling.
- **Evidence:** `package.json` line 18; imports from `@supabase/auth-helpers-nextjs` in 3 files
- **Recommendation:** `npm uninstall @supabase/auth-helpers-nextjs && npm install @supabase/ssr`. Update imports: `createServerClient` from `@supabase/ssr`, `createBrowserClient` from `@supabase/ssr`
- **Effort:** M | **Risk:** MED
- **Acceptance Criteria:** Auth flow works end-to-end (login, session refresh, tenant resolution, impersonation)

### Finding 3.3: Error Messages Leak Internal Details
- **Severity:** MEDIUM
- **Category:** Security
- **Files Affected:** Multiple API routes
- **Description:** Several routes return `error.message` directly to the client, which can expose stack traces, database error messages, or internal paths.
- **Evidence:** Pattern: `catch (error) { return NextResponse.json({ error: error.message }, { status: 500 }) }`
- **Recommendation:** Return generic error messages to client; log detailed errors server-side via `monitor.trackError()`
- **Effort:** S | **Risk:** LOW

### Finding 3.4: Zoning Route LIVE Tier Always Fails
- **Severity:** MEDIUM
- **Category:** Architecture
- **Files Affected:** `src/app/api/zoning/route.ts`
- **Description:** The LIVE tier implementation throws an error intentionally: `throw new Error('ArcGIS Endpoint DS-005 unverified - using fallback')`. This means zoning data always falls through to CACHED or MOCK.
- **Evidence:** zoning/route.ts line 26: `throw new Error('ArcGIS Endpoint DS-005 unverified')`
- **Recommendation:** Either implement the ArcGIS REST query (requires `ARCGIS_CLIENT_ID`) or document this as a known limitation in the LIVE tier status
- **Effort:** M | **Risk:** LOW

### Finding 3.5: Three-Tier Fallback — Well Implemented
- **Severity:** INFO
- **Category:** Architecture
- **Files Affected:** `src/lib/utils/fallback.ts`, flights/route.ts, zoning/route.ts, suburbs/route.ts
- **Description:** `fetchWithFallback<T>()` is a clean generic implementation that returns `{ data, tier, source, year, timestamp }`. Routes correctly wire LIVE→CACHED→MOCK chains.
- **Evidence:** fallback.ts — 79 lines, well-typed, proper error handling at each tier
- **Recommendation:** No change needed — this is a strong pattern

### Finding 3.6: No OpenAPI Specification
- **Severity:** LOW
- **Category:** Maintainability
- **Files Affected:** All API routes
- **Description:** No machine-readable API documentation exists (no OpenAPI/Swagger spec). This makes API discovery and contract testing difficult.
- **Recommendation:** Consider `next-swagger-doc` or manual OpenAPI YAML for external consumers
- **Effort:** M | **Risk:** LOW

### Finding 3.7: Middleware Rate Limiting Is In-Memory Only
- **Severity:** LOW
- **Category:** Architecture
- **Files Affected:** `src/middleware.ts`
- **Description:** Rate limiting uses an in-memory `Map` which resets on serverless cold starts and doesn't work across Vercel edge instances. Effective for local dev but not production.
- **Evidence:** middleware.ts lines 13-26: `const rateLimitMap = new Map<string, { count: number; resetAt: number }>()`
- **Recommendation:** For production, use Vercel KV or Upstash Redis for distributed rate limiting
- **Effort:** M | **Risk:** LOW
