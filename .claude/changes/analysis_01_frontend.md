## Findings

### Finding 1.1: MapLibre Ref Guard & Cleanup — Correct
- **Severity:** INFO
- **Category:** Architecture
- **Files Affected:** `src/components/map/MapContainer.tsx`
- **Description:** MapContainer uses `forwardRef` + `useImperativeHandle` with proper ref guard (`if (!mapContainer.current || mapInstance) return`) and cleanup (`map.remove()` in useEffect return). CSS imported at component level.
- **Evidence:** Lines 84-123 — single init with cleanup. Line 21: `import 'maplibre-gl/dist/maplibre-gl.css'`
- **Recommendation:** Consider importing MapLibre CSS in `layout.tsx` per CLAUDE.md §5 instead of component level for consistency.
- **Effort:** S | **Risk:** LOW

### Finding 1.2: CesiumJS Dynamic Import — Correct but Large Bundle
- **Severity:** MEDIUM
- **Category:** Performance
- **Files Affected:** `src/components/DashboardScreen.tsx`, `src/components/map/CesiumViewer.tsx`
- **Description:** CesiumJS is loaded via `next/dynamic` with `ssr: false`. SpatialView wraps both MapLibre 2D and Cesium 3D with mode switching. CesiumJS is ~40MB which significantly impacts bundle size.
- **Evidence:** DashboardScreen.tsx line 25: `const SpatialView = dynamic(() => import('./map/SpatialView'), { ssr: false })`
- **Recommendation:** Consider lazy-loading Cesium only when 3D mode is activated; currently loaded on every dashboard render.
- **Effort:** M | **Risk:** LOW
- **Acceptance Criteria:** Cesium bundle not loaded until user clicks "3D mode"
- **Rollback:** Revert to current eager dynamic import

### Finding 1.3: No React Error Boundaries Around Map Components
- **Severity:** MEDIUM
- **Category:** Architecture
- **Files Affected:** `src/components/DashboardScreen.tsx`, `src/components/map/SpatialView.tsx`
- **Description:** MapLibre and CesiumJS can crash (WebGL context loss, memory pressure on mobile). No error boundaries wrap these components.
- **Evidence:** DashboardScreen.tsx lines 237-240 — SpatialView rendered directly without error boundary wrapper
- **Recommendation:** Add React error boundary with fallback UI (static map image or "Reload" button) around SpatialView
- **Effort:** S | **Risk:** LOW

### Finding 1.4: Zustand Stores Not Found in Source
- **Severity:** LOW
- **Category:** Architecture
- **Files Affected:** `package.json` (zustand dependency), all components
- **Description:** Zustand v5 is listed as a dependency but no Zustand store files (`create()`) were found in the source. State management is done via React useState in DashboardScreen.tsx and React Context (TenantProvider).
- **Evidence:** `package.json` line 33: `"zustand": "^5.0.0"` but `grep -r "create(" src/ --include="*.ts"` finds no Zustand stores
- **Recommendation:** Either implement Zustand stores for shared state (map state, flight data, analysis results) or remove the unused dependency
- **Effort:** S | **Risk:** LOW

### Finding 1.7: `userScalable: false` in Viewport — Accessibility Concern
- **Severity:** MEDIUM
- **Category:** Compliance
- **Files Affected:** `src/app/layout.tsx`
- **Description:** `userScalable: false` prevents users from zooming the page, which is a WCAG 2.1 Level AA failure (Success Criterion 1.4.4 Resize Text).
- **Evidence:** layout.tsx line 27: `userScalable: false`
- **Recommendation:** Remove `userScalable: false` or set to `true`
- **Effort:** S | **Risk:** LOW
- **Acceptance Criteria:** Users can pinch-zoom on mobile devices
