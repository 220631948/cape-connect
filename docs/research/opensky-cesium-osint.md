## 1. Overview
This report details the integration of live ADS-B flight data into a CesiumJS-based OSINT dashboard.

## 2. OpenSky Network API
- **State Vectors:** The `GET /states/all` endpoint provides real-time state vectors (ICAO24, callsign, lat/lon, altitude, velocity).
- **Polling Constraints:** Authenticated users (free tier) have a rate limit of 5-second polling intervals. Non-authenticated users are restricted to 10 seconds.
- **State Data:** Includes barometric altitude, geometric altitude, and ground speed.

## 3. CesiumJS Rendering Strategy
- **CZML (Cesium Language):** The recommended standard for time-dynamic entity updates. A backend (Node/Python) should poll OpenSky and stream CZML packets to the frontend.
- **Interpolation:** CesiumJS's `HermitePolynomialApproximation` is used to smoothly interpolate aircraft positions between state vector updates, preventing "teleporting" aircraft.
- **Entity Updates:** Individual aircraft are mapped to `Entity` objects. 3D models (`.glb`) can be dynamically assigned based on aircraft category (e.g., small drone, large airliner).

## 4. Architecture & Scale
- **Rate Limit Strategy:** A backend caching layer (Redis) should poll OpenSky once per interval and serve the combined state to multiple connected clients to avoid exhausting API credits.
- **Service Quotas:** 100/day (anonymous), 1,000/day (registered authenticated); commercial license required for production multi-tenant deployment.
- **Multitenant Isolation:**
  - Flight layers filtered by geographic bounding boxes (Cape Town standard: `[18.0, -34.5, 19.5, -33.0]`).
  - Per-tenant API key injection and request namespacing to prevent cross-tenant billing leaks and ensure POPIA-compliant data boundaries.
