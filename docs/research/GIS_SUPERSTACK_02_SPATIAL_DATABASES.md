# GIS_SUPERSTACK_02_SPATIAL_DATABASES

This document evaluates the spatial database and storage layer(s) for the CapeTown GIS Hub (`capegis`), identifying the optimal backends for operational, analytical, and planetary-scale workloads.

## Domain Goal

Identify the right spatial storage layer(s) for the application's scale, query complexity, and multi-tenancy requirements.

## 1. PostGIS (The Operational Backbone)

**Status**: ✓ VERIFIED [Standard] | **Recommended**: P0

PostGIS remains the "PostgreSQL for GIS" and the default operational store for `capegis`. All tenant-scoped tables with RLS must reside in PostGIS.

### Key Operators & Indexing

- **Operators**: `ST_Intersects`, `ST_DWithin`, `ST_Contains`, `ST_Buffer`.
- **Indexing**: GiST (Generalized Search Tree) for spatial, B-Tree for `tenant_id`.
- **Pattern**: Always use `ST_SetSRID(..., 4326)` for storage and `ST_AsMVT` for vector tile generation.

### Multi-Tenancy (RLS)

```sql
ALTER TABLE parcels ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON parcels
  USING (tenant_id = current_setting('app.current_tenant')::uuid);
```

## 2. DuckDB-Spatial (The Analytical Engine)

**Status**: ✓ VERIFIED [Source: https://duckdb.org/] | **Recommended**: P1 (NEW)

DuckDB-Spatial v1.5.0 (March 2026) is the "SQLite for Analytics" with elite geospatial support. It is 10x-50x faster than PostGIS for GeoJSON/Parquet analytical queries.

### Integration Strategy: Analytical Offloading

1.  **Export**: Export large datasets (e.g., GV Roll) from PostGIS to GeoParquet.
2.  **Load**: Load GeoParquet into DuckDB-Wasm in the browser or DuckDB-Spatial in an Edge Function.
3.  **Query**: Perform high-speed joins and aggregations on-the-fly.

### Sample Query

```sql
SELECT zoning, sum(market_value)
FROM read_parquet('valuation_2022.parquet')
WHERE ST_Intersects(geom, ST_GeomFromText('POLYGON(...)'))
GROUP BY zoning;
```

## 3. BigQuery GIS (Planetary Scale)

**Status**: ✓ VERIFIED | **Recommended**: P2 (Enterprise/Cloud Only)

Use for petabyte-scale analytics and cross-referencing with global datasets (e.g., Earth Engine datasets).

### Limitations

- Not suitable for real-time app rendering due to high latency (>1s).
- Costly for small, frequent queries.

---

## 4. ClickHouse Geo (Real-time Telemetry)

**Status**: ✓ VERIFIED | **Recommended**: P2 (Flight Tracking)

Excellent for high-concurrency, high-throughput ingestion of time-series spatial data like flight tracks from the OpenSky Network.

### Pattern: Trajectory Storage

Store flight points as a sequence of (lat, lng, alt, timestamp) and use ClickHouse spatial functions to calculate distance or check if a flight passed through a risk zone.

---

## Recommendation & Implementation Path

- **Operational Data**: Keep in **PostGIS** on Supabase.
- **On-the-fly Analytics**: Implement **DuckDB-Spatial** for the Valuation Analysis Dashboard.
- **Flight Data**: Cache OpenSky signals in PostGIS `api_cache` for now, but move to **ClickHouse** if telemetry volume exceeds 1M points/day.
