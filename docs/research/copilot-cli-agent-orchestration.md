## 1. Overview
This report outlines the best practices for steering "vibecoding" agents within the GitHub Copilot CLI environment, specifically tailored for the Cape Town GIS project.

## 2. Configuration Standards
- **AGENTS.md:** The canonical governance document. While a backup exists at `AGENTS.md.bak`, the core rules are consolidated in **CLAUDE.md** for this project.
- **.agent.md Files:** Located in `.github/agents/` (e.g., `infra-agent.agent.md`, `orchestrator.agent.md`). These provide the specialized "brains" for different subsystems.
- **Skill Files:** Located in `.claude/skills/` (e.g., `spatial_validation`, `tile_optimization`). These define modular capabilities that agents can "activate."

## 3. Steering Protocols
- **Documentation-First Development:** A non-negotiable rule in `CLAUDE.md`. No code is written until a PRD or technical plan is established to prevent "hallucinated architecture."
- **Hooks:** The project utilizes `.claude/settings.json` hooks (e.g., `H-line-limit`, `H-badge-remind`) to enforce code standards during the agent's "Plan -> Act" cycle.
- **Multi-Agent Orchestration:** Validated by the "WorldView" project (Sidhu, 2025), which successfully used 8 simultaneous agents. The Cape Town GIS project formalizes this into a "Fleet" architecture with clear boundaries and handoff protocols.

## 4. Reproducible Research Pipelines
- **Vibecoding Safety Rules:** Embedded in `.github/copilot-instructions.md`. These rules force the agent to verify claims against the `docs/` folder before making assumptions.
- **Lock Files:** A recommended pattern for parallel agents writing to shared memory (e.g., shared memory files).
