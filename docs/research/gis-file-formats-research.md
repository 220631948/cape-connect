# ArcGIS and QGIS File Format Support

> **TL;DR:** Browser ingestion strategy: GeoJSON/KML native in CesiumJS; Shapefiles via `gdal3.js` (WASM) or `shpjs`; GeoPackage via `sql.js`; GeoTIFF via `geotiff.js`. `.qgz`/`.gdb`/`.mxd`/`.aprx` require server-side processing. All formats must be reprojected to EPSG:4326 via `proj4js`. Large datasets (LiDAR, massive rasters) offload to Cesium ion.

## 1. Overview
This report investigates the ingestion and visualization of legacy and modern geospatial formats within a browser-based CesiumJS environment.

## 3. Technical Translation (2026 Standards)

| Format | Library / API | Role |
| :--- | :--- | :--- |
| **Shapefile** | `shpjs` | Browser-side parsing to GeoJSON |
| **GeoPackage** | `sql.js` (WASM) | Vector geometry extraction |
| **File Geodatabase** | `ArcGIS REST API` | Remote query/conversion |
| **Reprojection** | `proj4js` | Normalization to **EPSG:4326** |
| **3D Rendering** | `CesiumJS 1.139+` | **KHR_gaussian_splatting** support |

## 3. Browser Visualization Patterns
- **WASM Parsing:** `gdal3.js` (GDAL compiled to WebAssembly) allows the browser to run `ogr2ogr` commands locally, enabling users to drag-and-drop Shapefiles and convert them to GeoJSON without a backend.
- **CRS Auto-Detection:** `proj4js` is the standard for handling Coordinate Reference Systems. Browser-side auto-detection of `.prj` files is possible using `gdal3.js`.
- **Coordinate Systems:** CesiumJS strictly requires WGS84 (EPSG:4326) or Web Mercator (EPSG:3857). All ingested files must be reprojected client-side using WASM tools.

## 4. Implementation Recommendations
1. **Frontend Ingestion:** Use `gdal3.js` for lightweight vector conversion.
2. **Heavy Data:** Point clouds (`.las`) and massive imagery (`.geotiff`) should be offloaded to a backend pipeline or Cesium ion for tiling.
3. **WASM Constraints:** Large datasets may exceed browser memory limits (typically 2GB-4GB) during WASM processing.
